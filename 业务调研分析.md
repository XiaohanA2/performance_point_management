# 个金系统业务调研分析报告

## 一、现状问题

**当前系统**：按4种角色配置任务数（manager/lobby_manager/elastic_counter/counter_manager）

**业务实际**：每个员工有独立的任务数配置，同角色、同支行的员工任务数也可能不同

## 二、业务实施逻辑（基于Excel分析）

### 2.1 组织架构
- **4个考核组**：大堂经理组、弹性柜员、客户经理组、柜员组
- **员工属性**：姓名、网点（支行）、所属组
- **任务分配**：到具体员工个人，非按角色统一

### 2.2 必选业务（8项，权重100分，封顶160分）

| 业务 | 单位 | 权重分 | 封顶分 | 计分方式 |
|------|------|--------|--------|----------|
| 1. 基金有效客户数 | 户 | 10 | 20 | 贡献度 |
| 2. 借记卡绑卡 | 万 | 10 | 20 | 贡献度 |
| 3. 期交保险销量 | 万 | 20 | 40 | 贡献度 |
| 4. 个人养老金开户 | 户 | 10 | 20 | 贡献度 |
| 5. 折效收单商户 | 户 | 10 | 20 | 贡献度 |
| 6. 信用卡分期 | 元 | 15 | 20 | 贡献度 |
| 7. 信用卡新增激活 | 户 | 10 | 15 | 贡献度 |
| 8. 企微福利官 | 户 | 5 | 7.5 | 贡献度 |

**贡献度公式**（组内相对排名）：
```
完成数 >= 组平均：
  得分 = 权重分 + (完成数 - 平均值) / (最大值 - 平均值) × 权重分 × 0.4

完成数 < 组平均：
  得分 = 权重分 - (平均值 - 完成数) / (平均值 - 最小值) × 权重分 × 0.4

最终得分 = min(得分, 封顶分)
```

### 2.3 加分业务（12项，封顶40分）

| 业务 | 计分规则 | 封顶 |
|------|----------|------|
| 1. 代发工资拓户 | 每10户得5分 | 无 |
| 2. 新发基金销量 | 按单支基金任务，单支封顶5分，未破零扣2分 | 5分/支 |
| 3. 贵金属销量 | 0.1分/10克 | 无 |
| 4. 智慧客 | 0.5分/户 | 2分 |
| 5. 互联网场景商户 | 1分/户 | 2分 |
| 6. 信用卡弹屏率 | 2分/10% | 无 |
| 7. 非本网点保险双录 | 1笔1分 | 5分 |
| 8. 拉新 | 人工评分 | 3分 |
| 9. 促活 | 人工评分 | 3分 |
| 10. 基金保险持证 | 固定分 | 5分 |
| 11. 上报业绩人员 | 人工评分 | 3分 |
| 12. 执行力 | 人工评分 | 6分 |

**加分总分** = min(各项加分之和, 40)

### 2.4 总分计算
```
必选总分(封顶160) + 加分总分(封顶40) = 共总得分(封顶200)
挂钩考核金额 = 共总得分 × 系数（封顶5万元）
```

### 2.5 任务分配特征

**个性化程度高**：
- 基金任务数分布：3/8/10/15/20户（5档）
- 保险任务数分布：5/5.5/7/8/9/11/13/14/15/18/20万（11档）

**分配规律**：
- 同支行同角色员工：任务数通常相同
- 同支行不同角色：任务数不同（如营业厅：客户经理20户，柜员15户）
- 不同支行同角色：任务数可能不同（根据网点规模/业绩能力）

**示例**：
```
西洋支行：
  陈耀（大堂）：基金15 保险18
  张婷（大堂）：基金15 保险18
  张潇蓓（柜员）：基金10 保险9

金峰支行：
  周仕培（客户经理）：基金20 保险20
  江信星（柜员）：基金15 保险11
```

## 三、系统改造方案

### 3.1 数据模型调整

#### 员工表（新增）
```javascript
// opendb-pf-employees
{
  _id: string,
  name: string,           // 姓名
  branch: string,         // 支行
  group: string,          // 考核组：lobby_manager/elastic_counter/manager/counter
  role: string,           // 角色（兼容旧字段）
  status: 'active'|'inactive',
  createdAt: Date
}
```

#### 员工任务配置表（新增）
```javascript
// opendb-pf-employee-tasks
{
  _id: string,
  employeeId: string,     // 员工ID
  taskId: string,         // 业务ID（pf_tasks._id）
  target: number,         // 个人任务数
  month: string,          // 月份 '2026-01'
  createdBy: string,
  createdAt: Date
}
```

#### 业务配置表（调整）
```javascript
// pf_tasks（保持现有结构，移除targetByRole）
{
  _id: string,
  name: string,
  type: 'required'|'bonus',
  unit: string,
  scoreConfig: {
    type: 'benchmark'|'linear'|'manual',
    weightScore: number,    // 必选业务权重分
    maxScore: number,       // 封顶分
    unitPrice: number       // 加分业务单价
  },
  // 移除 targetByRole
  order: number,
  status: 'active'|'inactive'
}
```

### 3.2 管理后台功能

#### 员工管理
- 员工列表：支持按支行/组别筛选
- 新增/编辑员工：姓名、支行、考核组
- 批量导入：Excel导入员工信息

#### 任务分配
**筛选维度**：
- 支行（下拉多选）
- 考核组（下拉多选）
- 员工姓名（搜索）

**批量配置**：

2. 筛选员工
3. 对每项必选业务配置任务数
4. 支持批量设置（选中员工统一赋值）
5. 支持单个调整

**配置界面**：

### 3.3 计分逻辑调整

#### 提报时校验
```javascript
// 提报时检查是否有任务配置
const employeeTask = await db.collection('opendb-pf-employee-tasks')
  .where({
    employeeId: userId,
    taskId: taskId,
    month: currentMonth
  })
  .get()

if (!employeeTask.data.length) {
  throw new Error('未配置任务数，请联系管理员')
}
```

#### 积分计算
```javascript
// 必选业务：从employee-tasks读取个人任务数
const target = employeeTask.target

// 贡献度计算：组内排名（不变）
const groupSubmissions = await getGroupSubmissions(group, taskId, month)
// ... 现有贡献度公式
```

### 3.4 兼容性处理

**数据迁移**：
1. 从现有submissions提取员工信息生成员工表
2. 从pf_tasks.targetByRole生成初始employee-tasks配置

**渐进式切换**：
```javascript
// 优先读取个人配置，回退到角色配置
let target = await getEmployeeTarget(userId, taskId, month)
if (!target) {
  target = task.targetByRole[userRole] // 兼容旧逻辑
}
```

## 五、关键差异点

| 维度 | 当前系统 | 业务实际 |
|------|----------|----------|
| 任务分配 | 按4种角色 | 按员工个人 |
| 配置粒度 | 角色级 | 员工级 |
| 灵活性 | 低（同角色统一） | 高（个性化） |
| 管理方式 | 修改角色配置 | 筛选员工批量配置 |
| 数据结构 | targetByRole | employee-tasks表 |
